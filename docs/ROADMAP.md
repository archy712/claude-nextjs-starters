# Notion 견적서 뷰어 MVP 개발 로드맵

노션에 입력한 견적서를 클라이언트가 별도 가입 없이 고유 링크로 확인하고 PDF로 다운로드할 수 있게 하는 웹 서비스.

## 개요

**Notion 견적서 뷰어**는 노션으로 견적서를 관리하는 1인/소규모 사업자를 위한 견적서 공유 서비스로 다음 기능을 제공합니다:

- **노션 동기화 (F001)**: 관리자가 버튼 클릭 한 번으로 노션 데이터베이스의 견적서를 앱 DB로 가져와 저장/갱신
- **견적서 목록 & 링크 관리 (F002, F005)**: 동기화된 견적서를 대시보드에서 확인하고, 견적서별 고유 링크를 즉시 복사
- **로그인 없는 견적서 확인 & PDF (F003, F004)**: 클라이언트가 고유 링크로 접속해 로그인 없이 견적서를 확인하고 PDF로 다운로드

**핵심 사용자 여정**: 첫 접속(인증 체크) → 로그인/회원가입 → 관리자 대시보드(동기화·목록·링크 복사) → (링크 전달) → 견적서 상세 페이지(클라이언트, 비로그인, PDF 다운로드)

## 개발 워크플로우

1. **작업 계획**

- 기존 코드베이스를 학습하고 현재 상태를 파악
- 새로운 작업을 포함하도록 `ROADMAP.md` 업데이트
- 우선순위 작업은 마지막 완료된 작업 다음에 삽입

2. **작업 생성**

- 기존 코드베이스를 학습하고 현재 상태를 파악
- `/tasks` 디렉토리에 새 작업 파일 생성
- 명명 형식: `XXX-description.md` (예: `001-setup.md`)
- 고수준 명세서, 관련 파일, 수락 기준, 구현 단계 포함
- **API/비즈니스 로직 작업 시 "## 테스트 체크리스트" 섹션 필수 포함** — 아래 4개 카테고리를 빠짐없이 Playwright MCP 시나리오로 작성:
  - 정상 플로우 테스트 (happy path)
  - 에러/예외 케이스 테스트 (실패 응답, 유효성 검증 실패, 인증 오류 등)
  - 엣지 케이스 테스트 (빈 값, 경계값, 중복 데이터 등)
  - 기존 기능 회귀 테스트 (연관된 다른 플로우에 영향이 없는지)
- 예시를 위해 `/tasks` 디렉토리의 마지막 완료된 작업 참조. 예를 들어, 현재 작업이 `012`라면 `011`과 `010`을 예시로 참조.
- 이러한 예시들은 완료된 작업이므로 내용이 완료된 작업의 최종 상태를 반영함 (체크된 박스와 변경 사항 요약). 새 작업의 경우, 문서에는 빈 박스와 변경 사항 요약이 없어야 함. 초기 상태의 샘플로 `000-sample.md` 참조.

3. **작업 구현**

- 작업 파일의 명세서를 따름
- 기능과 기능성 구현
- **API 연동 및 비즈니스 로직 구현 시 Playwright MCP로 테스트 수행은 선택이 아닌 필수 절차**
- 각 단계 후 작업 파일 내 단계 진행 상황 업데이트
- 구현 완료 후 Playwright MCP를 사용한 E2E 테스트를 반드시 실행 (정상/에러/엣지 케이스 모두 포함)
- **테스트를 통과하기 전까지 해당 Task/단계는 완료(✅)로 표시하지 않음**
- 테스트 실패 시 원인 분석 후 수정 → 통과할 때까지 재테스트 반복
- 테스트 통과를 확인한 후에만 다음 단계로 진행
- 각 단계 완료 후 중단하고 추가 지시를 기다림

4. **로드맵 업데이트**

- 로드맵에서 완료된 작업을 ✅로 표시

## 기술 스택

- **Framework**: Next.js 15.5.3 (App Router + Turbopack) / React 19.1.0 / TypeScript 5
- **Styling/UI**: TailwindCSS v4 · shadcn/ui (new-york) · Radix UI · Lucide
- **Forms**: React Hook Form + Zod + Server Actions
- **External**: `@notionhq/client` (노션 연동) · `@react-pdf/renderer` (PDF 생성)
- **Backend/DB**: Supabase (인증 + PostgreSQL)
- **Deploy**: Vercel / 패키지: npm

## 현재 상태 요약

- 라우트 골격 존재: `/`(랜딩), `/login`, `/signup`, `/dashboard`, `/quote/[slug]`
- shadcn/ui 공통 컴포넌트 설치 완료 (button, card, input, form, dialog, sonner 등)
- 로그인/회원가입 폼은 정적 UI 상태 (Zod + React Hook Form 미연결)
- 대시보드/견적서 상세는 플레이스홀더 상태
- `@notionhq/client`, `@react-pdf/renderer`, `@supabase/*` 미설치 · DB 모델·Server Actions 미구현

  - 관련 기능: `F002`, `F005` (UI 레벨)

### Phase 3: 핵심 기능 구현

- **Task 007: Supabase 연동 및 데이터베이스 구축** - 우선순위
  - `@supabase/supabase-js`, `@supabase/ssr` 설치 및 클라이언트/서버 클라이언트 설정
  - `quotes`, `quote_items` 테이블 및 인덱스/제약(`slug` unique) 마이그레이션 적용
  - 견적서 조회용 데이터 액세스 레이어 작성 (`src/lib/db/quotes.ts`)
  - 견적서 상세 페이지 더미 데이터를 실제 DB 조회로 교체 (`/quote/[slug]`)
  - 관련 기능: `F011`, `F003`
  - **Playwright MCP 통합 테스트 (4개 카테고리 필수)**:
    - 정상: 유효한 slug로 견적서 상세·품목 정상 조회/렌더링
    - 에러/예외: DB 연결 실패·조회 예외 시 에러 처리, 존재하지 않는 slug → `not-found`
    - 엣지: 품목 0개 견적서, 초장문 견적서명/금액 경계값, 특수문자 slug
    - 회귀: Phase 2에서 완성한 상세 페이지 UI/레이아웃이 실제 데이터 연동 후에도 정상
  - 위 테스트 전부 통과를 확인한 후에만 완료(✅) 처리

- **Task 008: 관리자 인증 시스템 구현**
  - Supabase Auth 기반 회원가입/로그인/로그아웃 Server Actions 구현
  - 로그인/회원가입 폼을 Server Actions에 연결 (성공 시 리디렉션, 실패 시 에러 표시)
  - 루트(`/`) 및 `/dashboard` 미들웨어 기반 인증 가드 (미인증 → `/login`)
  - 세션 관리 및 최초 1회 관리자 가입 정책 반영
  - 관련 기능: `F010`
  - **Playwright MCP E2E 테스트 (4개 카테고리 필수)**:
    - 정상: 회원가입 → 로그인 → 대시보드 진입 → 로그아웃 전체 플로우
    - 에러/예외: 잘못된 자격증명 로그인 실패, Zod 유효성 검증 실패, 미인증 상태로 `/dashboard` 접근 시 `/login` 리디렉션
    - 엣지: 중복 이메일 가입 차단, 빈 값/공백 입력, 최초 1회 관리자 가입 정책 경계
    - 회귀: Task 007에서 연동한 견적서 상세(비로그인 공개 경로)가 인증 가드 도입 후에도 접근 가능한지
  - 위 테스트 전부 통과를 확인한 후에만 완료(✅) 처리

- **Task 009: 노션 데이터 동기화 구현**
  - `@notionhq/client` 설치 및 Notion API 클라이언트 설정 (env 키 연결)
  - 지정 노션 데이터베이스 조회 → `Quote`/`QuoteItem` 매핑 파서 구현
  - "노션 동기화" Server Action 구현 (upsert 기준 `notion_page_id`, `synced_at` 갱신)
  - 신규 견적서에 추측 불가능한 고유 `slug` 토큰 자동 발급
  - 대시보드 목록을 실제 DB 데이터로 교체
  - 관련 기능: `F001`, `F002`, `F005`, `F011`
  - **Playwright MCP 테스트 (노션 API는 목업/샌드박스, 4개 카테고리 필수)**:
    - 정상: 동기화 실행 → 목록 갱신 → 신규 slug 발급 → 링크 복사 플로우
    - 에러/예외: 노션 API 오류/타임아웃, 인증 실패, 매핑 파싱 실패 시 에러 처리 및 부분 실패 롤백
    - 엣지: 빈 데이터베이스(0건) 동기화, 동일 `notion_page_id` 재동기화 시 upsert(중복 미생성)·`synced_at` 갱신, 대량 견적서
    - 회귀: Task 008 인증 가드 및 Task 007 상세 조회가 동기화 로직 추가 후에도 정상
  - 위 테스트 전부 통과를 확인한 후에만 완료(✅) 처리

- **Task 010: 고유 링크 발급·복사 및 접근 제어**
  - 견적서별 고유 링크 URL 생성 로직(`/quote/{slug}`) 및 클립보드 복사 실동작 연결
  - 유효하지 않은/만료 slug 접근 시 안내 페이지 처리 확정
  - 클라이언트 상세 페이지의 비로그인 공개 접근 및 관리자 전용 경로 분리 검증
  - 관련 기능: `F005`, `F003`
  - **Playwright MCP 테스트 (4개 카테고리 필수)**:
    - 정상: 링크 복사 → 새 세션(비로그인)에서 링크 접근 → 견적서 노출
    - 에러/예외: 유효하지 않은/만료 slug 접근 시 안내 페이지, 비로그인 상태의 관리자 전용 경로(`/dashboard`) 차단
    - 엣지: 추측 가능한 임의 slug 접근 차단, 삭제된 견적서 링크 접근
    - 회귀: Task 008 인증 가드가 클라이언트 공개 경로 접근을 막지 않는지, Task 009 발급 slug와의 일관성
  - 위 테스트 전부 통과를 확인한 후에만 완료(✅) 처리

- **Task 010-1: 핵심 기능 통합 테스트**
  - Playwright MCP를 사용한 전체 사용자 여정 E2E 테스트 — 4개 카테고리 전부 커버:
    - 정상: 가입 → 로그인 → 동기화 → 링크 복사 → 비로그인 상세 접근 여정
    - 에러/예외: 동기화 실패, 인증 만료, API 오류 응답 핸들링
    - 엣지: 잘못된 slug, 빈 목록, 품목 0개, 중복 동기화
    - 회귀: Task 007~010에서 구현한 기능이 이번 통합 과정에서 깨지지 않았는지 확인
  - API 연동(Notion/Supabase) 및 비즈니스 로직 검증
  - 하나라도 실패 시 해당 Task로 돌아가 원인 수정 후 전체 재테스트 → 전부 통과를 확인한 후에만 완료(✅) 처리
